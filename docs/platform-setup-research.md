# Platform Setup Function ‚Äî Feasibility Research

## Goal

Extend the platform plugin protocol so that platform plugins can expose a **`setup()`** method.

When a platform advertises `capabilities.setup = true`, the omni-dashboard shows a purple **"Setup"** button on `/platforms` at the left of the "Reinstall" button (for installed/enabled platforms).

Clicking "Setup" sends `POST /api/plugins/{name}/setup` with the current plugin config. The backend routes the request to the platform plugin, which runs the setup logic and returns success/error. Errors are generated by the platform code and shown to the user as-is ‚Äî the backend/dashboard doesn't need to interpret them.

## Current Architecture

Mattermost plugin `main.rs` method dispatch (line 733):

```rust
let response = match request.method.as_str() {

    "deliver"       => ...
    "edit_message"  => ...
    "delete_message" => ...
    "react"         => ...
    _ => make_error(req_id, -1, &format!("Unknown method: {}", request.method)),
};
```

### Current Setup Script (`mm-setup.py`)

2. Logs into Mattermost via REST API
3. Creates users if they don't exist
4. Creates team
5. Adds members to team
6. Creates channel
7. Adds members to channel
8. Creates bot account + personal access token
9. Updates `.env` with `MATTERMOST_ACCESS_TOKEN` and `MATTERMOST_CHANNEL_IDS`

## Proposed Architecture

### 1. Backend: Add `capabilities.setup` flag

In `omniagent/src/plugin/mod.rs`:

```rust
pub struct PluginCapabilities {
    pub inbound: bool,
    pub outbound: bool,
    #[serde(default)]
    pub setup: bool,  // NEW ‚Äî when true, platform has a setup() method
}
```

The mattermost plugin's `handle_initialize` would return:
```rust
"capabilities": {
    "inbound": true,
    "outbound": true,
    "setup": true
}
```

### 2. Backend: Add `POST /api/plugins/{name}/setup` endpoint

In `omniagent/src/server/plugins.rs`, add to `plugin_router()`:

```rust
.route("/api/plugins/{name}/setup", post(setup_plugin_handler))
```

The handler does:
1. Load plugin manifest ‚Äî check `capabilities.setup == true`
2. Raise 400 if the platform doesn't advertise setup
3. Read the current plugin config from YAML (merged with defaults from `plugin.json`)
4. **Approach A (one-shot):** Spawn the plugin binary with `{"method":"setup", "params": config}` over stdin/stdout, wait for JSON-RPC response, return result (success or JSON-RPC error).
5. **Approach B (live plugin):** Send a `"setup"` request to the running platform plugin subprocess, wait for response, return result.
6. On success, optionally update the YAML config with any new values returned by setup (e.g., a new access token).
7. On error, return the error object as-is ‚Äî the frontend shows the error message to the user.

**Recommendation: Approach A (one-shot)** ‚Äî simpler, no lifecycle coupling, no risk of disrupting the running plugin.

#### Error Propagation

The backend does NOT interpret setup errors. The flow is:

```
Platform plugin ‚Üí JSON-RPC error {code, message} ‚Üí backend ‚Üí HTTP 400 {success: false, error: "..."} ‚Üí frontend toast
```

If the Mattermost plugin's `handle_setup` finds `setup_team` is missing, it returns:
```json
{"id": 1, "error": {"code": -1, "message": "Missing required config: setup_team ‚Äî use \"$env:MM_SETUP_TEAM\" or set it in the plugin config"}}
```

The backend wraps this into an HTTP response:
```json
{"success": false, "error": "Missing required config: setup_team ‚Äî use \"$env:MM_SETUP_TEAM\" or set it in the plugin config"}
```

The dashboard shows it as a toast. **No backend changes needed per-platform** ‚Äî the error source code lives entirely in the platform plugin.

#### Timeout Strategy

Setup can take 30+ seconds. Options:

| Option | Pros | Cons |
|--------|------|------|
| **Sync with long timeout** (e.g., 120s) | Simplest implementation; one request/response | Browser may timeout; no progress feedback |
| **Async: return immediately, poll** | Good UX; status updates | More complex; need status endpoint |
| **Async: WebSocket progress** | Real-time updates | Overengineered for setup |

**Recommendation**: Start with sync + 120s timeout. If the setup takes too long, the backend can be changed to async later without any frontend changes (the frontend just needs a loading spinner either way).

### 3. Mattermost: Add `handle_setup` to the plugin

New method in the protocol dispatch (`main.rs`):

```rust
"setup" => {
    if let Some(params) = request.params {
        let config: SetupParams = serde_json::from_value(params)
            .map_err(|e| make_error(req_id, -1, &format!("Invalid setup params: {}", e)))?;
        handle_setup(req_id, config).await
    } else {
        make_error(req_id, -1, "Missing params for setup")
    }
}
```

`SetupParams` struct:
```rust
struct SetupParams {
    setup_team: Option<String>,    // required for setup
    setup_channel: Option<String>, // required for setup
    bot_user: Option<String>,      // required for setup
    admin_user: Option<String>,     // optional
    admin_password: Option<String>, // optional
    test_user: Option<String>,      // optional
    test_password: Option<String>,  // optional
}
```

`handle_setup` would:
1. Validate required fields ‚Äî return error if missing (e.g., "Missing required config: setup_team")
2. If `admin_user`/`admin_password` provided ‚Üí try to login as admin (for elevated operations). Otherwise use the bot token (which is already authenticated via env).
3. Create team (`POST /api/v4/teams`) if not exists
4. Create channel (`POST /api/v4/channels`) if not exists
5. Create users if not exist:
   - `admin_user` (if provided) ‚Üí `POST /api/v4/users` + set `system_admin` role
   - `test_user` (if provided) ‚Üí `POST /api/v4/users`
   - `bot_user` ‚Üí `POST /api/v4/users` if not exists
6. Add all users to team
7. Add all users to channel
8. Create bot account (`POST /api/v4/bots`) for `bot_user`
9. Create/refresh personal access token (`POST /api/v4/users/{id}/tokens`)
10. Return success with:
    - `team_id`, `channel_id`
    - `bot_token` (the new personal access token)
    - `user_ids` map

#### New MattermostClient API methods needed

| Method | HTTP | Endpoint |
|--------|------|----------|
| `create_user()` | POST | `/api/v4/users` |
| `create_team()` | POST | `/api/v4/teams` |
| `add_team_member()` | POST | `/api/v4/teams/{id}/members` |
| `create_channel()` | POST | `/api/v4/channels` |
| `add_channel_member()` | POST | `/api/v4/channels/{id}/members` |
| `create_bot()` | POST | `/api/v4/bots` |
| `create_user_token()` | POST | `/api/v4/users/{id}/tokens` |
| `get_user_tokens()` | GET | `/api/v4/users/{id}/tokens` |
| `update_user_password()` | PUT | `/api/v4/users/{id}/password` |

These are straightforward ‚Äî they follow the same pattern as the existing `create_post()`, `get_me()`, etc.

**Key design note**: All 7 fields have `required: false` in the schema because they're not required for the *platform to run* ‚Äî only for the *setup action*. ### 5. YAML Env Var References (`$env:`)

In the `plugins.yaml` (or the platform's YAML registration at `platforms.yml`), the default config values reference env vars via `$env:` prefix:

```yaml
# platforms.yml (in omni-stack or data_dir)
platforms:
  mattermost:
    enabled: true
    config:
      setup_team: "$env:MM_SETUP_TEAM"
      setup_channel: "$env:MM_SETUP_CHANNEL"
      bot_user: "$env:MM_BOT_USERNAME"
      admin_user: "$env:MM_USERNAME"
      admin_password: "$env:MM_USER_PASSWORD"
      test_user: "$env:MM_TEST_USERNAME"
      test_password: "$env:MM_TEST_PASSWORD"
```

This is the SAME pattern already used by the config system ‚Äî e.g., `"$env:MATTERMOST_ACCESS_TOKEN"` in the current `resolved_env`. The `$env:` prefix is resolved at runtime from the process environment.

**The user can override these in the dashboard.** When they change values in the plugin config form and click Save, the backend writes the new values to the YAML `config` block (overriding the `$env:` defaults). On the next "Setup" click, the new values are used.

### 6. Frontend: Purple Setup Button

In `src/pages/platforms.ts`, `renderPlatformsPage()`:

```typescript
// Insert before Reinstall button: left of Reinstall, only when installed + has setup
${p.source !== "built-in" && !p.needs_build && p.manifest?.capabilities?.setup ? `
  <button type="button" class="plugin-setup-btn" style="background:rgba(139,92,246,0.15);border:1px solid rgba(139,92,246,0.3);border-radius:6px;padding:0.25rem 0.5rem;cursor:pointer;font-size:0.75rem;color:var(--accent-purple);">Setup</button>
` : ""}
```

In `wirePlatforms()`, add:
```typescript
document.querySelectorAll(".plugin-setup-btn").forEach((btn) => {
  btn.addEventListener("click", async () => {
    const card = (btn as HTMLElement).closest(".card") as HTMLElement;
    const pluginName = card?.getAttribute("data-plugin-name");
    if (!pluginName) return;

    const originalText = btn.textContent || "Setup";
    btn.textContent = "Setting up...";
    (btn as HTMLButtonElement).disabled = true;

    try {
      await apiPost(`/plugins/${encodeURIComponent(pluginName)}/setup`, {});
      (window as any).showToast?.("Setup complete", "success");
      void loadPlatforms();
    } catch (e) {
      // Error message comes directly from the platform plugin
      (window as any).showToast?.(
        "Setup failed: " + (e instanceof Error ? e.message : "Unknown error"),
        "error",
      );
      btn.textContent = originalText;
      (btn as HTMLButtonElement).disabled = false;
    }
  });
});
```

**Order in the toolbar**: Setup button inserted BEFORE the Reinstall button in the HTML source, so it appears left of Reinstall in the flex row.

### How It Works End-to-End

1. User navigates to `/platforms` ‚Üí sees Mattermost card with purple "Setup" button (left of Reinstall)
2. User expands config, fills in `setup_team: "omni"`, `setup_channel: "general"`, saves
3. User clicks "Setup" ‚Üí frontend calls `POST /api/plugins/mattermost/setup`
4. Backend loads `plugin.json`, sees `capabilities.setup = true`, reads current config from YAML
5. Backend spawns `mattermost-platform` binary with `{"method":"setup", "params":{setup_team:"omni",...}}` over stdin
6. The plugin validates required fields ‚Üí if missing, returns error (shown as toast)
7. Plugin runs setup: creates team, channel, users, bot, token
8. Plugin returns `{"team_id": "...", "channel_id": "...", "bot_token": "..."}`
9. Backend optionally updates YAML config with new values (e.g., `channel_ids`)
10. Frontend shows "Setup complete" toast, reloads platforms page

### Risk Areas

| Risk | Mitigation |
|------|------------|
| **Long setup blocks HTTP** | Start with 120s timeout; if needed, switch to async with polling |
| **Plugin binary not compiled** | Use the same compile-if-needed logic as install/reinstall |
| **Config writes to .env** | The setup may need to update `.env` with new token + channel IDs ‚Äî backend can do this after setup returns |
| **Duplicate runs** | Setup should be idempotent (create-if-not-exists for all resources) |

### Migration from Python `mm-setup.py`

The existing Python script does 8 things. All map directly to Mattermost REST API endpoints that the Rust `MattermostClient` can call:

| Python step | Rust equivalent | Difficulty |
|-------------|-----------------|------------|
| Read env vars | `std::env::var()` | Trivial |
| Login | Reuse existing `get_me()` (bot token already authed) | Trivial |
| Create users | New `create_user()` client method | Easy |
| Create team | New `create_team()` client method | Easy |
| Add team members | New `add_team_member()` client method | Easy |
| Create channel | New `create_channel()` client method | Easy |
| Add channel members | New `add_channel_member()` client method | Easy |
| Create bot + token | New `create_bot()` + `create_user_token()` methods | Easy |
| Update .env | Return token + channel IDs to backend; backend writes .env | Backend task |
